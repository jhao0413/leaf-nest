# Leaf Nest 云端化与移动端离线规划

## Summary

将当前“浏览器本地 SQLite 单机阅读器”重构为“云端主数据 + Web 在线优先 + 安卓端后续离线扩展”的架构，目标支持登录、跨设备同步、AI 问书、自部署和 NAS 场景。

已确定的决策：
- 后端框架：`Hono`
- 主数据库：`Postgres`
- ORM / Schema / Migration：`Drizzle ORM + drizzle-kit`
- 文件存储：`S3 / MinIO` 兼容
- 认证：`Better Auth`
- 部署方式：`Docker` 自部署优先
- 服务形态：首版 `单体 API`
- Web 策略：`在线优先`，当前不做浏览器离线能力
- 安卓策略：后续支持 `手动下载书籍后离线阅读`
- 兼容策略：`不兼容` 现有本地 SQLite 数据，直接按新模型重建

## Key Changes

### 1. 系统边界重定义

- 服务端成为唯一主数据源，负责用户、书籍元数据、阅读进度、标注、对象存储映射和后续 AI 相关数据。
- Web 端不再维护完整本地数据库，也不再承担离线真相源职责；浏览器仅保留必要的会话态和界面偏好。
- EPUB 原文件必须上传到服务端对象存储，Web 阅读时按需从服务端获取。
- 安卓端离线能力作为后续扩展，通过独立的移动端本地文件缓存和同步机制实现，不要求当前 Web 版本先实现一套浏览器离线数据库。

### 2. 后端能力设计

首版单体 Hono 服务包含以下模块：

- 认证模块：基于 Better Auth，首版只支持邮箱密码登录，并使用 Drizzle 适配数据库表。
- 书籍模块：书籍元数据管理、上传登记、内容访问授权、删除。
- 文件模块：对接 S3/MinIO，负责 EPUB/封面对象上传、读取、签名访问。
- 阅读数据模块：阅读进度、高亮、批注的读写。
- AI 接口模块：围绕书籍、章节、批注、检索结果暴露问答输入数据。
- 数据访问层：统一使用 Drizzle 定义 schema、关系和 migrations，保持对 Postgres SQL 的直接控制。
- 后台任务先不独立拆 worker，但 API 设计和表结构预留后续异步任务扩展位。

### 3. 数据模型方向

服务端 Postgres 首版至少包含以下核心实体：

- `users`
- `sessions` / `accounts` / `verification` 等 Better Auth 所需表
- `books`
- `book_files`
- `reading_progress`
- `highlights`
- `ai_jobs` / `ai_conversations` / `ai_sources` 这类 AI 预留表

服务端数据库变更原则：

- 所有业务表和 Better Auth 相关表统一纳入 Drizzle schema 管理。
- 迁移使用 `drizzle-kit` 生成和执行，不引入 Prisma 或 Supabase 专有 schema 管理流程。
- 所有会在未来参与移动端同步的实体都保留 `updated_at`，同步删除的实体保留 `deleted_at` 扩展空间。
- 查询层优先保持贴近 SQL 的表达方式，便于后续实现移动端同步、冲突处理和 AI 检索相关查询。

客户端首版仅保留：

- 会话相关的轻量状态缓存
- 主题、语言、版式等纯本地偏好

安卓端后续离线阶段再引入：

- 本地 EPUB 文件缓存
- 本地全文索引
- 本地阅读进度镜像
- 本地高亮/批注镜像
- 待同步操作队列

### 4. Web 与移动端策略

- Web 端默认在线使用，不以断网可用为目标。
- Web 阅读器通过服务端 API 或签名地址获取 EPUB 内容并在内存中解析，不将书籍文件持久化到浏览器数据库。
- 后续安卓端支持用户显式下载书籍后离线阅读，并在恢复联网时同步进度和批注。
- 当前阶段必须在代码边界上为移动端预留仓储接口和文件访问抽象，但不在 Web 端先实现 Dexie/IndexedDB 离线层。

### 5. 存储与部署原则

- 文件接口统一按 S3 协议抽象，默认兼容 MinIO。
- Docker Compose 首版至少包含：`app`、`postgres`、`minio`。
- 不引入 Supabase 专有能力作为核心依赖。
- 所有业务能力默认面向自部署环境设计，SaaS 化仅作为后续可选部署形态。
- ORM 层默认使用 Drizzle，避免引入过重的数据访问抽象。

## Public Interfaces

需要新增或明确的对外接口类型：

- 认证接口：注册、登录、登出、会话获取
- 书籍接口：创建书籍记录、上传文件、列出书籍、获取详情、删除书籍
- 文件访问接口：获取封面或 EPUB 的短时访问地址，供 Web 在线阅读和后续安卓下载复用
- 阅读数据接口：同步进度、拉取进度、高亮与批注的增删改查
- AI 接口：针对书籍上下文发起问答、查询来源片段

需要统一的内部类型契约：

- `Book`
- `BookFile`
- `ReadingProgress`
- `Highlight`
- `AuthSession`
- `BookBinarySource`

## Test Plan

- 认证测试：邮箱密码注册、登录、登出、无效凭证、未登录访问受保护接口。
- 书籍测试：上传 EPUB、生成书籍记录、获取书籍列表、删除书籍、对象存储失败处理。
- Web 阅读测试：在线打开书籍、加载目录、恢复阅读进度、提交高亮与批注。
- 冲突测试：两端同时更新同一本书的进度和批注时，验证最终写入规则符合预期。
- 文件测试：MinIO 下上传、读取、签名访问、删除流程可用。
- AI 准备测试：AI 接口能读取书籍元数据、章节文本和批注上下文。
- 部署测试：Docker Compose 一键启动后，完成注册、上传、阅读、批注闭环。

## Assumptions

- 首版不迁移旧本地 SQLite 数据。
- 首版不做第三方 OAuth，只做邮箱密码。
- 首版不拆独立 worker，但表结构和模块边界要为后台任务预留扩展空间。
- 当前 Web 版本不做离线阅读，不实现浏览器本地书库持久化。
- 安卓离线阅读是后续阶段需求，当前仅预留仓储与文件访问抽象。
- 对象存储以 S3/MinIO 兼容为唯一目标，不额外适配 Supabase Storage 专有接口。
- ORM 选型固定为 `Drizzle ORM`，不再额外评估 Prisma。
